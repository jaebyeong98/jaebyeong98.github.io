# 1. Docker vs. Virtual Machine

## 가상 머신 (Virtual Machine)

- 가상 머신은 하이퍼바이저 (Hypervisor)라는 소프트웨어를 통해 물리적 서버 위에 여러 개의 게스트 OS를 실행함
- 하이퍼바이저는 물리 서버의 하드웨어를 가상화하여 각 가상머신이 자신만의 독립된 커널과 운영체제를 사용하는 것처럼 동작하도록 만듦
- 게스트 OS는 자신만의 커널 모드와 유저 모드가 존재하며, 이를 통해 완전히 격리된 독립적인 시스템처럼 작동함
- 게스트 OS 관점에서는 마치 물리 서버 위에서 자신만의 운영체제를 실행하는 것처럼 보이지만, 실제로는 하이퍼바이저가 중간에서 하드웨어 자원을 관리하고 스케줄링함
- 게스트 OS는 하드웨어와 직접 소통하지 않고, 하이퍼바이저를 통해 간접적으로 자원을 요청하고 사용함
- 이런 방식은 운영체제를 완전히 격리하지만, 별도의 운영체제 부팅 과정과 커널 관리가 필요하므로 자원 소모와 오버헤드가 큼

## 도커 (Docker)

- 도커는 컨테이너 (Container)라는 가상화 기술을 기반으로 동작하며, 각 컨테이너는 호스트 OS의 커널을 공유함
- 컨테이너 내부는 각기 독립적인 환경을 가지지만, 커널은 호스트 OS와 동일하며 별도의 게스트 OS가 필요 없음
- 도커는 애플리케이션 실행에 필요한 파일과 라이브러리만 포함하며, 운영체제 자체는 포함하지 않기 때문에 크기가 작음
- 컨테이너의 동작은 기본적으로 호스트 OS의 프로세스처럼 관리되며, 별도의 하이퍼바이저가 필요 없음
- 컨테이너는 OS를 완전히 가상화하지 않으므로, 부팅 속도가 빠르고 자원 소모가 적음

## 하이퍼바이저와 게스트 OS의 동작 방식

### **하이퍼바이저(Hypervisor)**:

- 물리 서버의 하드웨어를 가상화하고, 여러 가상 머신(VM)을 생성하고 관리하는 소프트웨어
- 하드웨어를 분리된 여러 가상 리소스로 나눠서, 각각의 가상머신이 독립적으로 실행될 수 있도록 지원함
- 하이퍼바이저의 두 가지 종류:
    1. **Type 1**: 하드웨어 위에 직접 설치되는 "베어메탈(Bare-metal)" 하이퍼바이저 (예: VMware ESXi, Xen)
    2. **Type 2**: 기존 OS 위에서 실행되는 소프트웨어 기반 하이퍼바이저 (예: VirtualBox, VMware Workstation)
- 하이퍼바이저는 가상머신 내의 게스트 OS가 하드웨어를 직접 제어하는 것처럼 보이게 만들어줌. 이 과정에서 CPU, 메모리, 네트워크 등의 자원을 스케줄링하고 스위칭함

### 게스트 OS:

- 가상머신 안의 게스트 OS는 자신만의 커널과 운영체제 모드를 가짐
- 하이퍼바이저 덕분에 게스트 OS는 자신이 물리 하드웨어를 직접 제어한다고 생각하지만, 실제로는 하이퍼바이저가 자원 요청을 처리함
- 게스트 OS를 부팅하려면 운영체제 커널을 초기화하고, 파일 시스템을 로드하며, 프로세스를 실행해야 하므로 시간이 더 소요됨

## 도커의 자원 관리와 컨테이너 부팅 시간

- 도커 컨테이너는 호스트 OS 커널을 공유하므로, 별도의 운영체제 부팅 과정이 필요 없음
- 컨테이너는 실행될 때 호스트 OS에서 새로운 프로세스를 생성하며, 이 프로세스는 일반적인 애플리케이션처럼 실행됨
- 따라서 컨테이너는 애플리케이션 실행에 필요한 라이브러리와 설정만 포함하며, 운영체제 전체를 포함하지 않으므로 크기가 작고 부팅이 빠름
- 예를 들어, 가상머신은 부팅할 때 게스트 OS 커널 초기화 → 시스템 서비스 시작 → 사용자 애플리케이션 실행의 단계를 거쳐야 하지만, 도커는 바로 애플리케이션 실행 단계로 넘어감

## 도커의 효율성과 최소화된 환경

- 도커는 호스트의 리소스를 직접 사용하며, 하드웨어 가상화가 필요 없기 때문에 오버헤드가 적음
- 도커 이미지는 최소화된 리눅스 환경을 제공하여, 컨테이너 크기를 줄이고 효율성을 높임
ex) Alpine Linux는 매우 작은 크기의 리눅스 배포판으로, 컨테이너 크기를 최소화하는 데 자주 사용됨
- 도커의 컨테이너는 애플리케이션 실행에 꼭 필요한 파일과 의존성만 포함하기 때문에, 필요한 용량이 적고 배포 속도가 빠름

## 도커 컨테이너와 프로세스 관리

- 호스트 OS는 도커 컨테이너를 일반 프로세스처럼 관리함
    - 각 컨테이너는 독립적으로 실행되지만, 실제로는 호스트 OS 커널 위에서 동작하는 프로세스에 불과함
    - 이를 통해 컨테이너는 호스트 OS와 자원을 공유하며, 성능 손실을 최소화함
- 도커는 cgroups (Control Groups)와 네임스페이스 (Namespaces)라는 리눅스 커널 기능을 활용하여 리소스 제한과 프로세스 격리를 구현함
    - cgroups: CPU, 메모리, 디스크 I/O와 같은 리소스를 제한하거나 할당하는 기능
    - 네임스페이스: 프로세스 ID, 네트워크, 파일 시스템을 격리하여 컨테이너가 서로 독립적인 환경처럼 보이게 함

| **구분** | **도커(Container)** | **가상머신(Virtual Machine)** |
| --- | --- | --- |
| **운영체제(OS)** | 호스트 OS의 커널을 공유하며, 별도의 게스트 OS가 없음 | 각 가상머신이 독립된 게스트 OS를 실행 |
| **커널 공유 여부** | 호스트 OS의 커널을 공유 | 각 게스트 OS가 자신만의 커널을 가짐 |
| **부팅 과정** | 애플리케이션만 실행하므로 부팅이 매우 빠름 | 게스트 OS 부팅(커널 초기화, 시스템 서비스 실행)이 필요하므로 시간이 더 걸림 |
| **리소스 관리** | 호스트 OS가 직접 자원을 관리(cgroups, namespaces 활용) | 하이퍼바이저가 자원을 가상화하여 관리 |
| **오버헤드** | OS를 가상화하지 않으므로 오버헤드가 적음 | 운영체제 전체를 가상화해야 하므로 오버헤드가 큼 |
| **애플리케이션 관리** | 호스트 OS의 일반 프로세스로 실행됨 | 가상머신 내부에서 실행되며, 독립적인 환경 제공 |
| **크기** | 운영체제를 포함하지 않아 컨테이너 크기가 작음 (수십 MB) | 게스트 OS를 포함하므로 크기가 큼 (수 GB 이상) |
| **격리 수준** | 애플리케이션 레벨에서 격리 (cgroups, namespaces 기반) | 운영체제 레벨에서 완전 격리 |
| **유연성** | 호스트 OS와 동일한 커널을 사용하는 애플리케이션에 적합 | 다양한 운영체제를 실행할 수 있음 |
| **주요 기술** | 리눅스의 cgroups, namespaces | 하이퍼바이저 (Type 1: ESXi, Xen / Type 2: VirtualBox 등) |
| **성능** | 호스트 OS 위에서 직접 실행되므로 성능 손실이 적음 | 하이퍼바이저를 거쳐야 하므로 성능 손실이 발생 |
| **사용 사례** | 경량화된 애플리케이션 실행, 마이크로서비스 아키텍처, CI/CD | 다양한 OS 실행, 테스트 환경 제공, 고도로 격리된 환경이 필요한 작업 |

# 2. 도커의 장점

1. 일관된 환경 제공
2. 자원 효율
3. 빠르고 유연한 배포
4. **마이크로서비스 아키텍처와의 시너지**: 독립적으로 배포와 확장이 가능한 작은 서비스 단위로 애플리케이션을 분리하면서 도커를 활용
